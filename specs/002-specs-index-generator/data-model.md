# Data Model — Specs Index Generator

Phase 1 of `/speckit.plan`. Defines input/output shapes for the script. **No SQL schema** — this is a pure-function tool: filesystem + gh API in → markdown out.

---

## Input shape (per spec folder)

```ts
type SpecFolderScan = {
  /** The directory name, e.g. "001-murajaah-scheduler" */
  dirName: string;

  /** The 3-digit prefix (or null for non-conforming directories) */
  nnnPrefix: string | null;

  /** The slug after the prefix, e.g. "murajaah-scheduler" */
  slug: string | null;

  /** Branch name read from spec.md frontmatter line "**Feature Branch**: `<name>`" */
  branchName: string | null;

  /** Whether each artefact exists in the folder */
  artefacts: {
    spec: boolean;            // spec.md exists
    plan: boolean;            // plan.md exists
    research: boolean;        // research.md exists
    dataModel: boolean;       // data-model.md exists
    quickstart: boolean;      // quickstart.md exists
    contracts: boolean;       // contracts/ directory exists and is non-empty
    tasks: boolean;           // tasks.md exists
  };

  /** Whether spec.md contains a "## Clarifications" section with at least one Q→A bullet */
  hasClarifications: boolean;

  /** PR-state lookup result (gh pr list --head <branchName>) */
  prState: {
    number: number | null;       // PR number, null if no PR
    state: 'open' | 'merged' | 'closed-unmerged' | 'none';
    url: string | null;
    closedAt: string | null;     // ISO 8601 UTC; null if not closed
  };
};
```

**Validation rules:**
- `dirName` MUST match the regex `^(\d{3})-([a-z][a-z0-9-]*)$` for the row to appear in INDEX.md. Non-matching directories are skipped silently (they may be in-progress directories from before /speckit.specify ran, or test fixtures).
- `branchName` is read from the spec.md `**Feature Branch**: \`<name>\`` line. If the line is missing, `branchName` is `null` and the row's "Branch" column shows `(unknown)` with a warning emitted to `automation_logs`.
- `prState.state === 'closed-unmerged'` AND `closedAt` older than 90 days → row is suppressed entirely from INDEX.md (per Q5 90-day window).

## Lifecycle status state machine

```
            +---------+
            |  Draft  |  ← spec.md only, no Clarifications section
            +----+----+
                 |  add Q→A bullet under ## Clarifications
                 ▼
          +-----------+
          | Clarified |  ← spec.md + Clarifications with ≥1 Q→A
          +-----+-----+
                |  /speckit.plan creates plan.md
                ▼
            +---------+
            | Planned |
            +----+----+
                 |  /speckit.tasks creates tasks.md
                 ▼
         +-------------+
         | Tasks-ready |
         +------+------+
                |  PR opened (gh state=open)
                ▼
        +--------------+
        | Implementing |
        +-+----------+-+
          |          |  PR merged          PR closed-not-merged
          ▼          ▼
    +---------+  +-----------+
    | Shipped |  | Abandoned | ← (90-day window; auto-disappears)
    +---------+  +-----------+

    Anywhere with no spec.md → Malformed (warning logged)
```

**Transitions are inferred from the input shape, not stored.** The script is stateless — every run derives the lifecycle from current filesystem + gh state.

## Output shape — INDEX.md format

The generated `specs/INDEX.md` has two sections:

```markdown
# FURQAN Specs Index

> Auto-generated by `scripts/generate-specs-index.ts`.
> Regenerated by husky pre-commit (on `specs/**/*.md` changes) and by an n8n nightly cron (03:00 UTC).
> **Do not edit by hand** — your changes will be overwritten on the next regen.

## Active specs

| # | Status | Branch | PR |
|---|--------|--------|----|
| [001-murajaah-scheduler](./001-murajaah-scheduler/spec.md) | Planned | `001-murajaah-scheduler` | [#221](https://github.com/drdeebtech/furqan/pull/221) |
| [002-specs-index-generator](./002-specs-index-generator/spec.md) | Tasks-ready | `002-specs-index-generator` | _(pending)_ |

## Abandoned (last 90 days)

_None._

---
*Run `npm run specs:index` locally to regenerate.*
```

**Output rules:**

1. The "Active specs" table includes statuses: `Draft | Clarified | Planned | Tasks-ready | Implementing | Shipped | Malformed`.
2. The "Abandoned (last 90 days)" section is a separate table; if empty, render `_None._` so the section is always present (avoids layout drift between regens).
3. Each row's first column is a link to the spec.md inside the folder.
4. PR column shows `[#NNN](url)` when known; `_(pending)_` when no PR exists yet.
5. Branch column shows the branch in inline-code formatting.
6. Sorted: Active by NNN ascending; Abandoned by `closedAt` descending.
7. **No timestamp in the file body.** Idempotency requires zero diff on no-state-change; a "last generated" line would always change. The file's `git log` shows regen history.

---

## Constraints

- **Idempotency** — running twice with no state change produces zero diff (FR-008). Sort order is deterministic; line endings normalised to `\n`; trailing whitespace trimmed; no embedded timestamps.
- **No external state** — the script keeps no DB, no cache file. Each run is fresh.

## Migration outline

No DB migration. The implementation PR adds:
- `scripts/generate-specs-index.ts` (new file)
- `scripts/__tests__/generate-specs-index.test.ts` (new file)
- `.husky/pre-commit` (new file, created by `npx husky init` then customised)
- `package.json` updates: `prepare` script, `specs:index` script, husky + lint-staged devDeps, lint-staged config block
- `specs/INDEX.md` (new file, generated by first run)
