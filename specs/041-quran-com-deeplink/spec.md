# Spec 041 — Quran.com deep-link on follow-up cards

**Issue:** #567
**Type:** Small client-side feature (link generation). **No** DB, **no** API key, **no** server work, **no** Quran text stored or generated.
**Status:** Architect draft — for owner review before build.

> Why no full speckit bundle (plan/tasks/data-model/contracts): this feature adds one pure
> function + one button. There is no data model, no API contract, no migration. A single spec
> with an embedded task list is proportionate; expand to the full bundle only if scope grows.

---

## 1. Problem & goal

Students need to hear canonical recitation and read the ayah text of their follow-up (homework)
ranges between sessions. Furqan's integrity rules (AGENTS.md §2) prohibit storing or generating
Quran text — so we **link out** to Quran.com's public, canonical pages instead of rendering any
scripture ourselves.

**Goal:** on each student follow-up item that has a surah:ayah range, show a
"استمع / Listen on Quran.com" button that opens the exact range on Quran.com, in a new tab,
with a reciter matching the student's `recitation_standard` when we can map it.

## 2. The three lenses

- **📖 Quran teacher (highest priority):** we render **zero** scripture — deep-link only, so
  integrity is preserved by construction. The one scholarly judgement is the **qari ↔ riwaya
  map** (which Quran.com reciter validly recites each `recitation_standard`). This is a fiqh/qira'at
  point → **flag for human/scholar verification; do not guess** (see Open Question OQ-2). Ship only
  the mappings a human confirms; fail soft on the rest.
- **🛠 Engineer:** pure client-side URL builder, no secrets, no network call from us. `rel="noopener
  noreferrer"` + `target="_blank"`. Fail-soft: unknown/null standard → link the range without a
  reciter param, never a broken link. Unit-test the builder.
- **🎓 Teaching-platform:** button sits on the student follow-up card, Arabic-first RTL, clear
  affordance, opens in a new tab so the student doesn't lose their place.

## 3. Design

**Where:** `src/app/student/follow-up/homework-list.tsx` — render the button per item **only when**
the item has a valid surah + ayah range (fields already exist: `surah`/`surah_from`/`surah_to`,
`ayah_from`/`ayah_to` or `ayah_start`/`ayah_end` — builder confirms the exact ones in code).

**New pure module:** `src/lib/quran/quran-com-link.ts`
- `buildQuranComUrl({ surah, ayahStart, ayahEnd, recitationStandard }): string`
- Validates surah/ayah against `src/lib/quran/ayah-counts.ts` (reuse — never trust raw range input).
- Appends the reciter param **only** if the standard maps to a verified reciter; otherwise omits it.
- Returns the range link regardless (fail-soft).

**Qari map:** a small constant `RECITATION_STANDARD_TO_QARI` (standard → Quran.com reciter id/slug),
**seeded only with owner/scholar-verified entries.** Start with the one safe default and leave the
rest unset until verified:
- `hafs` → Mishary Rashid Alafasy (Mishary recites Ḥafṣ ʿan ʿĀṣim) — the obvious default, still
  owner-confirm the exact Quran.com reciter id.
- `warsh`, `khalaf_hamzah`, (+ any other enum values) → **UNSET pending scholar verification.**

## 4. Open questions (resolve before/while building)

- **OQ-1 (verify, don't assume):** confirm Quran.com's **actual** URL/param format against the live
  site. The issue proposes `https://quran.com/[surah]/[ayah]?startVerse=[start]&endVerse=[end]` and a
  reciter param — the real param names (e.g. `startingVerse`/`endingVerse`, `reciter=`) must be
  checked on quran.com before we hardcode them. Negative control: build a URL, open it, confirm it
  lands on the exact range with the exact reciter.
- **OQ-2 (scholar):** the full `recitation_standard` → Quran.com reciter map. Only `hafs` is
  obvious. Warsh/Khalaf/etc. need a human who knows both the riwaya and Quran.com's reciter roster.
  **Flag for the owner / a qualified reviewer.**
- **OQ-3:** enumerate the complete `recitation_standard` value set (seen: `hafs`, `warsh`,
  `khalaf_hamzah`; confirm there are no others) so the map + fallback cover every case.

## 5. Acceptance criteria

- [ ] "استمع / Listen on Quran.com" button on each student follow-up item **that has a range**.
- [ ] Button links to Quran.com for that exact surah + ayah range (OQ-1 format verified live).
- [ ] Reciter param applied **only** for owner/scholar-verified standards; unmapped/null standard →
      valid range link with no reciter param (fail-soft, never broken).
- [ ] Opens in a new tab, `rel="noopener noreferrer"`.
- [ ] Correct in Arabic RTL — verified by browser screenshot, not assumed.
- [ ] No Quran text stored or generated by Furqan (deep-link only).
- [ ] Range validated against `ayah-counts.ts`; out-of-range → button hidden, not a bad link.

## 6. Tasks (build order)

1. `src/lib/quran/quran-com-link.ts` — `buildQuranComUrl` + `RECITATION_STANDARD_TO_QARI`
   (hafs-only seed). Validate range via `ayah-counts.ts`. Fail-soft on unknown standard.
2. `quran-com-link.test.ts` — unit tests: hafs → reciter param present; unknown/null → no reciter,
   still valid; out-of-range → throws/empty per contract; boundary ayahs.
3. Wire the button into `homework-list.tsx` (guard on range present), Arabic label, new tab, noopener.
4. **OQ-1** verify Quran.com URL format live; **OQ-2** get the scholar-verified qari map from owner,
   extend the constant.
5. RTL browser screenshot at the student follow-up route; vision-check the button + Arabic.

## 7. Non-goals

- No Quran text rendering, no audio hosting, no Quran.com API calls, no API key, no server route,
  no migration. No qari mapping shipped that a human hasn't verified.
