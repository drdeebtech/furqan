# Spec 026 — Guard `src/types/database.ts` against schema drift

**Status:** Backlog
**Created:** 2026-06-21
**Owner:** unassigned
**Risk:** Medium (96 importers; type-narrowing breakage if mishandled)

---

## Problem

The repo has **two generated-shaped Supabase type files**, and the more widely
used one is unguarded:

| File | Importers | Regenerated by | CI drift guard |
|------|-----------|----------------|----------------|
| `src/types/database.ts` | **96** | nothing in repo | ❌ none |
| `src/types/supabase.generated.ts` | 38 | `npm run db:types` | ✅ `.github/workflows/db-types-fresh.yml` |

`database.ts` carries the Supabase codegen marker (`__InternalSupabase` /
`PostgrestVersion: "14.5"`), so it is generated output — but no script
regenerates it and no CI job compares it to the live schema. If the production
schema changes, `database.ts` silently goes stale while 96 import sites keep
compiling against outdated types. This is a latent correctness risk (a column
type or enum value can drift without any signal).

The companion file `supabase.generated.ts` is correctly guarded but is the
*less*-used of the two.

## Why it wasn't fixed inline

This is a **decision**, not a mechanical edit. It needs a human call on
direction (Option A vs B below) and — for Option B — a coordinated 96-import
migration that warrants its own reviewed PR. It must not be done blind.

## Goal

Eliminate the unguarded-drift risk on `database.ts` with the lowest blast
radius that actually closes the gap.

## Options

### Option A — Guard in place (recommended, low risk)

Make `database.ts` a regenerated, CI-guarded artifact like its sibling.

1. Add an npm script that regenerates `database.ts` from the linked schema
   (mirror `db:types`, different output path), OR document that the two files
   are byte-identical and collapse one into a re-export of the other.
2. Add `src/types/database.ts` to the watched paths + the diff check in
   `.github/workflows/db-types-fresh.yml` (regenerate → `diff -q` → fail with a
   "run `npm run db:types:database` and commit" message).
3. No change to the 96 import sites.

**Pros:** ~10-line change, zero import churn, closes the gap immediately.
**Cons:** keeps two type files (line-count debt remains).

### Option B — Consolidate (higher risk, removes the duplication)

1. Confirm `database.ts` and `supabase.generated.ts` are content-equivalent
   (or reconcile any real differences — investigate first; do not assume).
2. Migrate all 96 `@/types/database` imports to `@/types/supabase.generated`
   (use `gitnexus_rename`, never find-and-replace — see AGENTS.md §7).
3. Delete `database.ts`.
4. The existing `db-types-fresh.yml` now covers the single remaining file.

**Pros:** one generated file, ~6,800 fewer lines, single guard.
**Cons:** 96-file diff; risk of silent type-narrowing changes if the two files
differ in any field. Requires the three-lens review (§1) and a fresh-apply
typecheck pass.

## Recommendation

Start with **Option A** — it removes the actual risk (silent drift) for almost
no blast radius. Schedule Option B separately only if the line-count duplication
becomes a real burden; gate it on first proving the two files are equivalent.

## Acceptance criteria

- [ ] A CI job fails a PR when `src/types/database.ts` diverges from the live
      schema (same mechanism as `db-types-fresh.yml`).
- [ ] A documented one-command path to regenerate `database.ts` locally.
- [ ] `npx tsc --noEmit` clean; `npm run test:unit` green.
- [ ] (Option B only) all 96 importers migrated via `gitnexus_rename`; no
      remaining `@/types/database` references; fresh-apply typecheck passes.

## Out of scope

- Any change to `supabase.generated.ts` or its existing guard.
- Schema migrations themselves (this is purely about type-file freshness).

## Three-lens check (AGENTS.md §1)

- 🛠 **Engineer:** the real risk is silent type drift across 96 import sites;
  Option A is the minimal correct guard, Option B trades risk for cleanliness.
- 📖 **Quran-integrity:** none — type plumbing only, no rendered text or
  `surah:ayah` data touched.
- 🎓 **Teaching-platform:** none directly; indirectly protects every typed read
  (student/teacher dashboards) from compiling against stale column types.
